## Detailed Evaluation

### Q1: "what is the limitation of self hosted version?" — 3/10 FAIL

**Golden answer expects:** Warm starts, auto-scaling, checkpoints, dedicated support missing; hardcoded limits (I/O packet length 128KB, alerts/schedules/team members/preview branches capped at 100M); configurable limits via env vars.

**Agent returned:** Generic infrastructure/ops differences — managing Postgres/Redis, SMTP not pre-configured, manual OAuth setup, manual image updates. None of the specific missing Cloud features were mentioned.

**Diagnosis:** The self-hosting overview page's limits/comparison table was either not indexed or its chunks didn't rank high enough. The agent confabulated reasonable-sounding but wrong content.

---

### Q2: "Does the waiting time of triggerAndWait and batchTriggerAndWait called within a trigger counts as computing time?" — 9/10 PASS

**Golden answer expects:** No, waiting does not count as compute. Parent task is checkpointed/paused. Reference to /wait page.

**Agent returned:** Correctly said `triggerAndWait` and `batchTriggerAndWait` do not consume compute time while waiting. Explained the parent task is "suspended" and removed from execution environment. Referenced v3 architecture and pricing docs.

**Minor miss:** Didn't name the `/wait` docs page specifically or mention the term "checkpointed."

### Q3: "run cron every 30 min s" — 9/10 PASS

**Golden answer expects:** `*/30 * * * *` cron pattern, `schedules.task()` code example, mention of timezone option, link to /tasks/scheduled.

**Agent returned:** Correct cron pattern, working `schedules.task()` code example with proper import, explained key properties (`id`, `cron`, `run`), linked to Scheduled Tasks documentation.

**Minor misses:** Imported from `@trigger.dev/sdk/v3` (golden uses `@trigger.dev/sdk`); didn't mention timezone option.

### Q4: "how to get the cost of the runs" — 3/10 FAIL

**Golden answer expects:** TRQL queries using `compute_cost` column in `runs` table, SQL example with ORDER BY/LIMIT, `prettyFormat(compute_cost, 'costInDollars')`, reference to /observability/query page.

**Agent returned:** Said documentation does not provide a specific API method to retrieve financial cost. Pointed to Usage Dashboard and Runs API generically. Completely missed TRQL, `compute_cost`, and the query/observability feature.

**Diagnosis:** The `/observability/query` docs page was either not indexed or the query expansion didn't map "cost of runs" to TRQL-related terms. Only 1 source was retrieved.

### Q5: "Can you show me the documentation page(s) for how to exit a task if certain criteria are met within my task?" — 4/10 PARTIAL FAIL

**Golden answer expects:** `AbortTaskRunError` — throw it to immediately fail the task and prevent retries. Code example with import. Reference to /errors-retrying page.

**Diagnosis:** The `AbortTaskRunError` section of the errors-retrying page was not surfaced by the search. Query expansion likely didn't map "exit a task" to "AbortTaskRunError."

## Root Cause Analysis

| Issue | Affected Qs | Likely Cause |
|-------|-------------|--------------|
| Missing chunks in index | Q1, Q4, Q5 | The self-hosting limits table, TRQL/query docs, and `AbortTaskRunError` section likely weren't indexed or ranked too low in hybrid search |
| Query expansion gaps | Q4, Q5 | `expandQuery()` doesn't map "cost of runs" to TRQL/compute_cost or "exit a task" to AbortTaskRunError — these are SDK-specific terms |
| Low source retrieval | Q1, Q4, Q6 | Only 1-4 sources used; BM25+vector top-8 RRF reranking misses key pages |
| LLM confabulation on thin context | Q1, Q5 | When RAG context is sparse, the LLM fills in generic advice instead of acknowledging gaps |

---

## Recommendations

1. **Audit index coverage** — Verify these pages are indexed with sufficient granularity:
   - `/self-hosting/overview` (the feature comparison / limits table)
   - `/observability/query` (TRQL, `compute_cost`, `prettyFormat`)
   - `/errors-retrying` (`AbortTaskRunError` section)

2. **Improve chunking for tables and code blocks** — Tables with specific limits and code examples containing API names (`AbortTaskRunError`, `compute_cost`) may be getting lost in large chunks. Consider smaller, overlapping chunks for reference-heavy pages.

3. **Expand query rewriting vocabulary** — Add domain-specific term mappings in `expandQuery()`:
   - "cost of runs" / "run cost" / "billing" -> TRQL, compute_cost, query, observability
   - "exit a task" / "stop retrying" / "abort task" -> AbortTaskRunError, errors-retrying
   - "self-hosted limits" -> self-hosting overview, warm starts, checkpoints, auto-scaling

4. **Increase retrieval depth** — Consider top-12 instead of top-8 results to catch more relevant chunks, especially for multi-faceted questions.

5. **Add a "low confidence" signal** — When the LLM receives thin context, prompt it to say "I couldn't find specific documentation on this" rather than confabulating generic advice.
