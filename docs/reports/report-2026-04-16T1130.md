Tested 6 golden QA pairs from `docs/support-golden-answers.csv` against the live Trigger Support chat agent at `localhost:3000`. Each question was submitted in a fresh chat session. 4. BM25 index on `section` column in addition to `content`

## Detailed Evaluation

### Q1: "what is the limitation of self hosted version?" — 9/10 PASS

**Golden answer expects:** Warm starts, auto-scaling, checkpoints, dedicated support missing; hardcoded limits (I/O packet length 128KB, alerts/schedules/team members/preview branches capped at 100M); configurable limits via env vars.

**Agent returned:** Comprehensive four-section answer:
1. **Feature Exclusions** — warm starts, auto-scaling, checkpoints, dedicated support (all four ✓)
2. **Technical and Architectural Limits** — **I/O Packet Length: Hardcoded to 128KB** ✓, ARM support missing, log retention never auto-deleted, `docker-provider` resource enforcement gap
3. **Operational Responsibilities** — production readiness, maintenance, Docker socket security risk
4. **Minimum Hardware Requirements** — 3+ vCPU, 6+ GB RAM

Cited 6 sources: feature comparison, caveats, limits, kubernetes overview, self-hosting overview, webapp requirements.

**Improvement over last run:** Now surfaces the 128KB I/O packet length limit that was previously missing. Also added operational detail (Docker socket, hardware requirements).

**Minor miss:** Still doesn't mention the 100M caps for alerts/schedules/team members/preview branches or that most limits are configurable via env vars.

### Q2: "Does the waiting time of triggerAndWait and batchTriggerAndWait called within a trigger counts as computing time?" — 9/10 PASS

**Golden answer expects:** No, waiting does not count as compute. Parent task is checkpointed/paused. Reference to /wait page.

**Agent returned:** Correctly said waiting does **not** count towards compute usage. Explained the parent task is **checkpointed** and remains in a paused state, does not consume compute resources. Key details:
- **Free Waiting** — parent task is checkpointed, you don't pay for compute during idle time
- **Usage duration** — `usage_duration` and `compute_cost` do not include waiting time
- **Checkpointing logic** — Similar to `wait.for()` or `wait.until()`, `triggerAndWait` offloads parent execution until result is ready

Cited 4 sources: paused execution overview, use waitpoints instead of polling, getting the run cost and duration, benefits of using Trigger.dev for marketing workflows.

**Same as last run** — consistently strong answer.

### Q3: "run cron every 30 min s" — 10/10 PASS

**Golden answer expects:** `*/30 * * * *` cron pattern, `schedules.task()` code example, mention of timezone option, link to /tasks/scheduled.

**Agent returned:** Both declarative and imperative approaches:
1. **Declarative Schedule** — `schedules.task()` with `cron: "*/30 * * * *"` ✓, syncs on `dev`/`deploy`
2. **Imperative Schedule (SDK)** — `schedules.create()` with `deduplicationKey` for dynamic per-user schedules

**Key Technical Details:**
- Cron syntax: standard 5-part, **does not support seconds** ✓
- Timezone: defaults to `UTC`, configurable via `timezone: "America/New_York"` ✓
- Payload: `timestamp`, `lastTimestamp`, `timezone`, `scheduledId`, `externalId`, `upcoming` (next 5 dates)
- Deduplication: `deduplicationKey` is **per project**, not per environment

Cited 1 source (declarative schedules). Answer is comprehensive and technically precise — full marks.

### Q4: "how to get the cost of the runs" — 9/10 PASS

**Golden answer expects:** TRQL queries using `compute_cost` column in `runs` table, SQL example with ORDER BY/LIMIT, `prettyFormat(compute_cost, 'costInDollars')`, reference to /observability/query page.

**Agent returned:** Three methods:
1. **From your backend (SDK)** — `runs.retrieve()` or `runs.list()` returning `costInCents`, `baseCostInCents`, `durationMs` with code example
2. **From inside a running task** — `usage.getCurrent()` for real-time cost data with `costInCents`, `totalCostInCents`
3. **Using TRQL (Query Dashboard)** — Full SQL: `SELECT run_id, task_identifier, compute_cost, triggered_at FROM runs WHERE compute_cost > 0 ORDER BY compute_cost DESC LIMIT 10` ✓

**Important Notes:** Exclusions (waiting, retries, pre-execution time), math functions (`round(compute_cost, 4)`, `ceil()`, `floor()`, `abs()`). Cited 2 sources.

**Minor miss:** Still doesn't mention `prettyFormat(compute_cost, 'costInDollars')`.

## Remaining Gaps

- **Q1:** 100M caps for alerts/schedules/team members/preview branches still not surfaced — likely in MDX table rows that get split across chunk boundaries
- **Q6:** `TRIGGER_SECRET_KEY` env var name not mentioned — the LLM has the context (it's in the retrieved chunks based on the `manual-setup` source badge) but doesn't surface it in the answer
- **Q4:** `prettyFormat(compute_cost, 'costInDollars')` not mentioned — may need dedicated chunk or stronger retrieval signal

These gaps are now at the LLM generation layer rather than the retrieval layer — the chunks are being retrieved (evidenced by source badges) but the model is choosing not to include every detail. Further improvement would require either prompt engineering (e.g., "always mention environment variable names") or structured extraction from retrieved chunks.
